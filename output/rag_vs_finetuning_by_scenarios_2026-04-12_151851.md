# RAG vs fine tuning для production: що краще в яких сценаріях

## Короткий контекст

У продакшені питання «що краще – RAG чи fine-tuning?» саме по собі некоректне, тому що ці підходи вирішують **різні класи проблем**:

- **RAG** (Retrieval-Augmented Generation) змінює **дані, які модель читає під час запиту**: джерела знань, їхню актуальність, персоналізацію. Ваги моделі лишаються незмінними.
- **Fine-tuning** змінює **поведінку моделі**: стиль, формат, стабільність відповідей, здатність до специфічних задач (класифікація, extraction). Знання з документів «вшити» в параметри можна, але це дорого, складно оновлювати й ризиковано з точки зору забування та регуляторики.

Тому практичне запитання звучить так: 
> «У моєму конкретному сценарії production: де джерело проблем – у **знаннях** чи у **поведінці/форматі** моделі, і які в мене обмеження по latency, вартості, приватності, оновлюваності знань?»

Нижче – розбір по типових сценаріях. Наприкінці є **чекліст із 8 запитань**, який допоможе швидко прийняти рішення для вашого кейсу.

## Сценарій a: Внутрішній пошук / knowledge base

**Use case:** пошук по Confluence, internal wiki, ticketing-системі, SOP, політиках.

### Що зазвичай краще

- **RAG-first**, часто + легкий fine-tuning під формат/тон, але не під знання.

### Чому

- Знання **динамічні** (оновлення політик, нові документи, зміни процесів). RAG дозволяє оновлювати індекс без retrain.
- Потрібні **посилання на джерела** й audit trail: легше, коли модель читає конкретні документи, а не «щось у вагах».
- Fine-tuning по сирих документах → дорожче, важко оновлювати, ризик «катастрофічного забування» інших здібностей моделі.

### Технічні нюанси

**RAG:**
- Структурно-орієнтоване chunking (заголовки, розділи, табличні блоки).
- Hybrid search (BM25 + embeddings) для поєднання лексичного та семантичного пошуку.
- Re-ranking (наприклад, cross-encoder/bge-reranker) для підвищення релевантності.
- ACL/permissions-фільтрація документів за користувачем/роллю.

**Fine-tuning (опційно):**
- Instruction/format tuning невеликим LoRA-адаптером: стабільний шаблон відповіді (наприклад, «Коротка відповідь + цитата + посилання»).
- Навчання на реальних діалогах: що вважати «хорошою» відповіддю.

### Коли вибір може змінюватися

- Якщо база **дуже стабільна** (регламент, що оновлюється раз на рік) і RAG-інфраструктуру важко підтримувати (on-device, без векторної БД), можна більше покластися на **fine-tuned компактну модель** з вшитим знанням.
- Якщо є жорсткі обмеження (air-gapped середовище, нульова толерантність до додаткових сервісів), іноді роблять «offline retrieval → fine-tune», але це компроміс, не best practice.

**Висновок по a:** для внутрішнього пошуку **RAG – базове рішення**; fine-tuning – для поведінки й формату, а не як заміна retrieval.

## Сценарій b: Чат-помічник поверх корпоративних документів

**Use case:** внутрішній «Copilot» (HR, IT-help, sales enablement).

### Що зазвичай краще

- **Гібрид:** RAG + легкий fine-tuning.

### Чому

- **Факти й контент** – через RAG (аналогічно до сценарію a).
- **Behavior & UX:** стабільний тон, дотримання політики безпеки (не видавати PII, не вигадувати поза джерелами), впорядкований формат – це вже **поведінка**, де fine-tuning корисний.
- IBM і AWS у своїх гайдах радять для enterprise-асистентів: RAG для знань, fine-tuning для policy/стилю/формату.

### Технічні нюанси

**RAG:**
- Multi-hop retrieval (кілька запитів/ітерацій для складних питань).
- Документні ACL, фільтрація по ролях/проєктах.

**Fine-tuning:**
- Instruction tuning / chat templates на базі історичних діалогів з правильними відповідями (з урахуванням RAG-контексту).
- LoRA/QLoRA для зміни поведінки без повного retrain.

### Коли вибір може змінюватися

- Маленький POC, низькі вимоги до якості → можна почати з «чистого» RAG і системних промптів.
- Після накопичення користувацьких діалогів з’являється підстава додавати **supervised fine-tuning** для підвищення стабільності та зменшення довжини промптів.

**Висновок по b:** для корпоративного асистента **гібрид практично завжди кращий**: RAG дає знання, fine-tuning – UX і policy.

## Сценарій c: Класифікація / маршрутизація / тегування

**Use case:** класифікація тикетів, route-to-team, intent detection, автотеги.

### Що зазвичай краще

- **Fine-tuning-first** (іноді навіть без RAG).

### Чому

- Завдання **стабільні**, класи відносно фіксовані.
- Важлива **консистентність і latency** (класифікація в real-time pipeline).
- Моделі не потрібно читати великі документи – RAG дає мало вигоди порівняно з гарно натренованим класифікатором.

### Технічні нюанси

- Supervised fine-tuning (CE по класах) на open LLM чи менших моделях.
- Parameter-efficient tuning (LoRA, adapters) + можлива дистиляція в 1–3B модель для продуктивного inference.
- RAG може іноді додаватися лише як пояснювальний шар («чому ми так класифікували»), але не як основний механізм.

### Коли вибір може змінюватися

- Якщо класи дуже динамічні, можна певний час триматися на **prompt-based** підході (LLM + rules), але при масштабі логічно переходити до fine-tuning.

**Висновок по c:** для класифікації, маршрутизації й тегування **fine-tuning зазвичай краще**; RAG – другорядний.

## Сценарій d: Генерація тексту в специфічному тоні бренду

**Use case:** маркетингові тексти, шаблони листів, відповіді підтримки в «голосі бренду».

### Що зазвичай краще

- **Fine-tuning-first** (style/instruction tuning, LoRA) + RAG для актуальних фактів/деталей продукту за потреби.

### Чому

- Проблема тут – **як** модель пише (tone, persona, style), а не «які факти знає».
- Prompt engineering дає стартове рішення, але при масштабі (тисячі текстів/день) fine-tuning зменшує довжину промптів, підвищує стабільність і знижує вартість.

### Технічні нюанси

**Fine-tuning:**
- Набір пар «сирий текст → відредагований копірайтером текст у брендовому стилі».
- LoRA-адаптери для різних брендів / персон.

**RAG (якщо треба факти):**
- Подає актуальні дані: акції, ціни, характеристики продуктів; стилем керує вже fine-tuned модель.

### Коли вибір може змінюватися

- Single-brand сценарій з невеликим обсягом – можна деякий час працювати на промптах без fine-tuning.
- Multi-brand SaaS або агенції → fine-tuning (або multi-adapter підхід) практично неминучий для контролю стилю.

**Висновок по d:** для бренд-тону **fine-tuning – основний інструмент**, RAG – для фактів.

## Сценарій e: Структуровані відповіді / extraction

**Use case:** витяг полів з інвойсів, контрактів, медичних записів; заповнення JSON; нормалізація сутностей.

### Що зазвичай краще

- **Fine-tuning-first**, часто в гібриді з RAG для контексту/довідників.

### Чому

- Головне – **коректний формат та consistency**, а не оновлюваність бази знань.
- Fine-tuned моделі значно краще тримають схему JSON/таблиць, ніж zero-shot/prompt-only.

### Технічні нюанси

**Fine-tuning:**
- Supervised tuning на парах `input → target JSON/fields`.
- Parameter-efficient tuning + constrained decoding (JSON schema, function calling) для зменшення помилок формату.

**RAG:**
- Корисний, якщо extraction залежить від **референсних таблиць** (словники кодів, product catalogs, нормативні довідники).
- Retrieval підтягує «канонічні» значення, модель мапить сирий текст → ці значення.

### Коли вибір може змінюватися

- Якщо багато різних типів документів без стабільних схем → на ранньому етапі можна почати з RAG + prompt-based extraction, збираючи дані для подальшого fine-tuning.

**Висновок по e:** для строгого extraction **fine-tuning – ключовий**, RAG – як допоміжний шар для контексту та довідників.

## Сценарій є: Low-latency edge / on-prem

**Use case:** локальні асистенти на edge-пристроях, закриті on-prem системи з обмеженим «залізом».

### Що зазвичай краще

- Для простих задач з жорстким latency: **fine-tuning + distillation** локальної моделі.
- Для on-prem enterprise з великими внутрішніми знаннями: **гібрид (локальний RAG + fine-tuned модель)**.

### Чому

- RAG додає vector DB/індекс, мережеві запити, кеші → **збільшує latency та складність**.
- Fine-tuned компактна модель працює без додаткових компонентів, що особливо важливо на edge.

### Технічні нюанси

**Fine-tuning:**
- QLoRA/LoRA + подальша quantization (4/8-bit) для запуску на CPU/мобільних SoC.
- Distillation у дуже малі моделі (1–3B) під вузький набір задач.

**RAG:**
- Можливий локальний in-process індекс (faiss/annoy/SQLite-based) з обмеженим розміром.

### Коли вибір може змінюватися

- Якщо знання **часто змінюються**, але є можливість тримати невеликий локальний індекс, комбінують:
  - маленьку fine-tuned модель (поведінка/формат);
  - локальний RAG-шар для даних.

**Висновок по є:**
- Edge-и з жорстким latency → **fine-tuning-first**.
- On-prem enterprise з багатими локальними знаннями → **гібрид**.

## Сценарій ж: Мультимовні сценарії

**Use case:** підтримка кількох мов, крос-мовний пошук, глобальні продукти.

### Що зазвичай краще

- **RAG-first для знань** + selective **multi-lingual fine-tuning** для стилю.

### Чому

- Багато foundation-моделей уже мультимовні.
- RAG дозволяє:
  - зберігати документи в оригінальних мовах;
  - робити cross-lingual retrieval по embeddings.
- Fine-tuning потрібен для:
  - єдиного брендовоготону у всіх мовах;
  - уточнення доменної термінології.

### Технічні нюанси

**RAG:**
- Мультимовні embeddings (bge-m3, LaBSE тощо).
- Language detection + routing (за потреби).

**Fine-tuning:**
- Один multi-lingual adapter або окремі adapters per language.
- Слідкувати, щоб не «перенавчити» модель на домінантній мові й не погіршити якість на інших.

### Коли вибір може змінюватися

- Якщо 1–2 основні мови й багато labeled data однією мовою → можна сильніше fine-tune під неї.
- Якщо багато мов із малою кількістю даних для більшості → **RAG + промпти**, мінімальний fine-tuning.

**Висновок по ж:**
- Для багатомовних знань, що оновлюються → **RAG-first**.
- Для єдиного стилю бренду → **hybrid: RAG + multi-lingual fine-tuning**.

## Зведений орієнтир по сценаріях

**RAG-first (базовий шар – RAG):**
- Внутрішній пошук / knowledge base.
- Чат-помічники поверх корпоративних документів (з подальшим додаванням fine-tuning).
- Регульовані домени (медицина, право, фінанси) – для фактів і цитувань.
- Мультимовні системи з великими й оновлюваними базами знань.

**Fine-tuning-first:**
- Класифікація, маршрутизація, тегування.
- Генерація тексту в тоні бренду.
- Структурований extraction у JSON/таблиці.
- Low-latency edge/on-device задачі з вузьким доменом.

**Hybrid-by-default:**
- Складні регульовані домени: RAG для фактів + fine-tuning для формату, safety, reasoning.
- Корпоративні асистенти з багатим контентом і високими вимогами до UX/стилю.
- Мультимовні системи, де потрібно поєднати широту знань із брендовим тоном.

## Чекліст: як вирішити для свого кейсу за 10 хвилин

Відповіді на питання нижче дають грубу рекомендацію щодо архітектури.

1. **Що вам потрібно змінити в першу чергу?**  
   - Знання/факти → RAG.  
   - Поведінку/формат/стиль → fine-tuning.

2. **Як часто змінюються релевантні знання (документи, норми, продукти)?**  
   - Часто (тижні/місяці) → RAG обов’язковий.  
   - Рідко → можна робити більш агресивний fine-tuning.

3. **Чи важливі посилання на джерела та можливість аудиту?**  
   - Так (регулятори, комплаєнс, внутрішні політики) → RAG.  
   - Ні → fine-tuning може бути основним.

4. **Чи потрібні строго структуровані відповіді (JSON/таблиці) з низьким error rate?**  
   - Так → fine-tuning (плюс schema validation), RAG – другорядний.  
   - Ні → RAG + промптинг може бути достатнім.

5. **Які вимоги до latency та інфраструктури?**  
   - Дуже строгі, edge/on-device, мінімум сервісів → fine-tuning компактної моделі, RAG – мінімальний або відсутній.  
   - Нормальні cloud/on-prem ресурси → hybrid c RAG.

6. **Обсяг та тип трафіку?**  
   - Багато однотипних запитів (класифікація, шаблонні відповіді) → fine-tuning-first.  
   - Різноманітні запити по великому корпусу документів → RAG-first.

7. **Чи багато мов / доменів треба підтримувати?**  
   - Так, багато мов/доменів, знання розпорошені → RAG-first, селективний multi-lingual fine-tuning.  
   - Один домен, 1–2 мови → можна більше інвестувати у domain fine-tuning.

8. **Чи готова команда підтримувати повний RAG-стек (векторні БД, індекси, моніторинг)?**  
   - Так → hybrid часто дасть найкраще співвідношення якості/гнучкості.  
   - Ні → простіше почати з fine-tuning невеликої моделі або з дуже простого RAG.

**Як інтерпретувати чекліст:**
- Якщо ≥5 відповідей тягнуть у бік динамічних знань, посилань на джерела, багатомовних/багатодокументних сценаріїв → **RAG-first + опційний fine-tuning**.
- Якщо ≥5 відповідей тягнуть у бік стабільних задач, strict форматів, low-latency/edge → **fine-tuning-first, RAG як опція**.
- Якщо відповіді змішані (регульований домен, складні задачі, і знання, і поведінка важливі) → **гібрид: fine-tuned модель поверх RAG**.

**Local knowledge base**
- retrieval-augmented-generation.pdf / p.1–4 – RAG як спосіб відділити оновлювані знання від параметрів моделі.
- large-language-model.pdf / p.13 – ризики галюцинацій, forgetting при fine-tuning.

**Web (вибірково)**
- IBM – «RAG vs. Fine-tuning», «RAG vs fine-tuning vs prompt engineering» (2024–2025): радить RAG для знань, fine-tuning для поведінки й стилю.
- AWS Prescriptive Guidance – «RAG vs fine-tuning for enterprise»: для регульованих доменів рекомендує RAG-first + обережний fine-tuning.
- Decision framework статті (Atlan та подібні) – пропонують запитаньний підхід до вибору між RAG, fine-tuning і промптингом.
- «The Ultimate Guide to Fine-Tuning LLMs» (arXiv, 2024) – описує, коли fine-tuning дає найбільший виграш (класифікація, структурований output, стиль).
- Роботи про catastrophic forgetting у LLM – показують ризики агресивного доменного fine-tuning і підтверджують необхідність обережного підходу.
